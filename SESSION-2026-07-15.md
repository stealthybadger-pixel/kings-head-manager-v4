## ThermoPro TP25H2 Bluetooth meat thermometer — DONE, shipped
- Reverse-engineered + verified against the device and the `daniel-corbett/thermopro-cli`
  source (decompiled from the official Android app). Protocol:
  service `1086fff0-3343-4817-8bb2-b32206336ce8`, write char FFF1, notify char FFF2.
  Sequence: subscribe FFF2 → wait 0.5s → write 12-byte handshake
  `01 09 8a 7a 13 b7 3e d6 8b 67 c2 a0` to FFF1 → device streams `0x30` frames ~1/sec.
  Decode: battery=byte2; probes at byte offsets 5-6/7-8/9-10/11-12, 2-byte custom BCD
  (sign bit 0x80, hundreds (b1&0x70)>>4, tens b1&0x0F, ones (b2&0xF0)>>4, tenths b2&0x0F);
  `ff ff` = probe unplugged. Full write-up in repo: `thermopro_protocol_guide.md`;
  standalone capture tool: `scripts/ble_probe_sniffer.html`.
- Built `src/hooks/useBleThermometer.ts` (mirrors useBleScale): connect, handshake,
  keepalive re-send every 5s, auto-reconnect around the device's ~15-20s idle drop.
  Wired into Food Temp Checks modal — tap a live probe reading to fill the temp field,
  user still presses Record (never auto-saved). Probe slots labelled **"Raw Meat" /
  "Cooked Item"** (first/second plugged port, in slot order — flip if it reads swapped).
- **Silent reconnect after refresh** via `navigator.bluetooth.getDevices()` (no chooser).
  Plus clean `gatt.disconnect()` on pagehide/beforeunload so the probe frees its single
  connection slot and keeps advertising — this removed the need to power-cycle it after a
  refresh. Confirmed working live by the user.
- **DEVICE SETUP REQUIREMENT (non-obvious, operational):** silent reconnect needs the
  Chrome flag `chrome://flags/#enable-web-bluetooth-new-permissions-backend` = Enabled,
  once per device/browser. Without it you get the chooser dialog on every refresh. Must be
  enabled on the **Android tablet** and the **MacBook** (both Chrome). **iOS is NOT
  supported at all** (Apple disables Web Bluetooth in Safari/Chrome) — would need the
  Bluefy browser app. A one-time dismissible in-app hint (localStorage key
  `probeFlagHintDismissed`) points users to the flag.

### Tesseract/OCR for equipment temps — investigated only, NOTHING BUILT
User asked whether there were repo notes about using Tesseract + an ice-bath/ice-point
calibration check for reading equipment (fridge) temps off a phone camera. Searched
thoroughly — current files, every branch (local + all ~18 remotes, several not previously
fetched), commit messages, and full history content search (`git log -S`, which catches
anything ever added *or removed*). **Zero matches anywhere, ever** for "tesseract" or any
ice-bath/ice-point calibration terminology. `EquipmentTempChecks.tsx` is a plain manual-entry
form today — no camera, no OCR, no calibration flow.

Follow-up clarified the actual ask: move *both* the Invoice Scanner and a new
phone-camera-reads-the-fridge-display feature off cloud API calls (both currently call
Gemini's vision API from `InvoiceScanner.tsx` / the recipe-scan feature in `Kitchen.tsx`,
needing an API key + internet + per-call cost). Assessed feasibility, built nothing:
- **Invoice scanning → poor fit for Tesseract.** Gemini is doing two jobs at once right now:
  reading text AND understanding an unstructured, per-supplier-varying invoice layout into
  structured line items (name/cost/pack size/unit/supplier, arbitrary count per invoice).
  Tesseract only does the first job — swapping it in would still need hand-written
  per-supplier layout-parsing logic to reconstruct line items from raw text, a much bigger
  and more fragile undertaking than it looks, with ongoing maintenance every time a supplier
  changes their invoice template.
- **Fridge-display reading → reasonable fit, with a caveat.** Reading a 2-4 digit number off
  a display is a much more contained problem than invoice parsing. Caveat: plain Tesseract is
  specifically known to be weak on seven-segment LCD/LED displays (the blocky digital-clock
  digits most fridge thermometers use) — its training data is tuned for normal printed fonts.
  Purpose-built seven-segment OCR approaches exist and do noticeably better on exactly this
  case; worth picking that over generic Tesseract if/when this gets built.
- Reusable groundwork already in the codebase for whenever this gets built: the recipe-scan
  feature (`Kitchen.tsx`) already uses `<input type="file" accept="image/*" capture="environment">`
  to force-open a phone's rear camera directly — the same pattern would work for a fridge-temp
  capture button.
- No OCR library is installed (checked `package.json`). Nothing in `EquipmentTempChecks.tsx`
  changed. Purely a feasibility read, at the user's explicit request not to action anything.
