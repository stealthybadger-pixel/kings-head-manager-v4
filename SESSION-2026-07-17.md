## Supplier Catalogue Data Flow Audit (investigation only, no code changed)
Traced Catalog.tsx's exact execution path and found it downloaded the **entire**
~3,500+-doc `supplierProducts` collection unconditionally on mount, before the user selected
a supplier or typed anything — confirmed with exact code references and a Firestore query
trace, not summarised. Identified `useSupplierProductsBySupplier` (a scoped `where()` query)
already existed in the codebase but had zero callers — dead code that was the actual fix.

## Firestore Optimisation Phase 3 — Catalog.tsx supplier-first loading (committed `68f7e8a`, live)
Implemented the audit's recommendation: Catalog now uses `useSupplierProductsBySupplier`
when a supplier is selected (real `where()` query, live-verified: only that supplier's docs
downloaded), and only falls back to the full-collection `useSupplierProducts()` when
searching *without* a supplier selected (preserves exact existing cross-supplier search
behaviour — deliberately did not touch the search algorithm itself, per explicit
instruction). `useSupplierProducts()` gained an optional `enabled` param (default `true`,
so `useCatalogCapture.ts`'s existing unconditional call is unaffected). Live-tested against
the emulator: idle state fires zero Firestore requests, supplier select fires exactly one
scoped `where(supplier==...)` request, search-without-supplier correctly falls back to the
full collection. **Known follow-up, not fixed**: `useCatalogCapture.ts` (behind the always-
mounted `CatalogCaptureModal`) still calls `useSupplierProducts()` unconditionally on every
login regardless of navigation — same class of bug as Catalog.tsx had, just in a different
file, out of this phase's stated scope.

## Catalogue Matcher tool (`wip/catalogue-matcher` branch — NOT merged, NOT deployed)

Originated from a ChatGPT conversation (pasted in by the user) redesigning an earlier
"canonical products" idea into something simpler: Pantry Ingredients are already canonical,
so supplier products just need a `SupplierProduct.ingredientId` link to the right one — no
new collection. Talked through the UX with the user before building: one product at a time,
Accept/Reject, on Accept pick-or-create the Pantry Ingredient link, plus a brand-name bulk
review checklist and category+supplier auto-reject rules (e.g. "no meat from Booker/Urban")
for blanket cleanup — the brand-checklist UI was **designed but not built**; the auto-reject
rule mechanism **was** built (see below).

**Deliberately kept fully separate from the main app** per explicit instruction after an
initial wrong turn (first attempt wired it into `App.tsx`'s nav/routing — reverted). Lives at
`tools/catalogue-matcher/` with its own Vite HTML entry point (`index.html` + `main.tsx`,
own auth-gated shell borrowing `AuthProvider`/`Login`), reachable only via
`npm run dev` → `/tools/catalogue-matcher/` — no nav link, and confirmed excluded from
`npm run build`'s `dist/` output (no `rollupOptions.input` entry for it).
`tsconfig.json`/`tailwind.config.js`/`vitest.config.ts` all extended to also cover `tools/`.

### What was built
- `SupplierProduct.ingredientId?: string` added to `src/types.ts` (additive schema field —
  the link the whole tool exists to populate).
- `tools/catalogue-matcher/CatalogueMatcher.tsx` — the review loop. Reuses existing app
  pieces rather than duplicating logic: `ItemPicker` for search, the standard
  `useSupplierProductMutations`/`useIngredientMutations` hooks (so every decision commits
  immediately via a normal mutation — no separate "session file" needed for crash-safety,
  since the emulator itself is the safe scratch copy).
- `tools/catalogue-matcher/rules.ts` — auto-reject rule framework. One rule live: **"no
  frozen items except peas and frozen fruit, no exceptions"** (exact user instruction),
  reusing the app's existing `Fruit` category keyword list from `ingredientAutofill.ts`
  rather than a hand-rolled list. Matched products are held out of the one-at-a-time queue
  entirely and surfaced as a single bulk "Review & Delete" action with a preview + confirm
  (not silent auto-delete). Live-tested: 63 real products correctly removed (frozen halal
  beef ribeye, frozen edamame beans, etc.), peas/fruit correctly spared.
- `tools/catalogue-matcher/suggest.ts` — **bug found and fixed via live testing**: the
  shared `findBestIngredientMatch` (used for the AI Suggestion panel) hard-rejects any
  ingredient with words the product name lacks. This app's pantry names are frequently
  "Item - Descriptor" (e.g. "Dill - Fresh"), so a single generic descriptor word was
  killing exact matches outright ("Dill" → "No confident match found"). Did **not** touch
  the shared matcher (used for real cheaper-option detection elsewhere, already tuned for
  that job) — instead added a separate, more lenient scorer scoped to this tool only,
  where extra words on either side count against confidence instead of disqualifying the
  candidate. Exported `areWordsFuzzyEqual` from `src/utils/matching.ts` (previously
  private) to reuse its fuzzy-word rules rather than duplicating them — zero behaviour
  change for that function's existing callers. Caught a real bug in the new lenient scorer
  via its own test suite (a short product name like "Dill" could slip under the slack cap
  against a completely unrelated ingredient) before it shipped — fixed by requiring at
  least one real word overlap. 3 new tests, `tools/**/*.test.ts` added to
  `vitest.config.ts`'s include so they actually run.
- Editable name before creating a new Pantry Ingredient — was immediate/uneditable
  (auto-derived from the product name) before; now opens a small edit panel
  (Create & Link / Cancel) so a bad auto-derived name can be fixed before committing.

### Workflow established for continuing this across sessions/machines
Firestore emulator data is **not reachable across machines** (localhost-only) and **not
persistent** across an emulator restart unless exported. Established:
`firebase emulators:export <path>` to snapshot, `--import=<path>` on next start to resume.
Hit and fixed a real Windows gotcha: exporting *inside* the actively-watched project folder
(`E:\Projects\kings-head-manager-v4`) failed with `EPERM: operation not permitted, rename`
— almost certainly VS Code's file watcher locking the temp folder mid-rename. Fixed by
exporting to `C:\emulator-snapshot`, outside the watched folder. User has since exported at
least twice (`C:\emulator-snapshot`, most recently after the name-edit feature landed).
