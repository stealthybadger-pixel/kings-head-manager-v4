## Gemini API — dedicated account + model retirement bug (the big one)
User set up a **separate Google Cloud project** (`khkm-gemini-scanner`) specifically to hold
a Gemini API key for Invoice Scanner + Kitchen's recipe-scan feature, isolating that billing
from the main Firebase project. Both features already read the key from the same
`localStorage` key (`geminiApiKey`), entered via a Settings field on the Invoice Scanner
page — no code change was needed for the account-separation part itself, just generating a
new key at aistudio.google.com/apikey (the classic AI Studio flow, not the newer Vertex AI
"Agent Platform" console, which requires service-account binding for Gemini/Vertex keys and
was a dead end) and pasting it into that field.

**Root cause found while testing the new key**: both features were hardcoded to
`gemini-1.5-flash`, which has been deprecated/retired by Google — every scan attempt was
returning a 404. Confirmed via web search this model is gone; current recommended stable
model is `gemini-3.5-flash`. Fixed in both files (commit `1d00857`).

### Follow-up: gemini-3.5-flash itself intermittently 503ing
After the model-name fix, image-based scans (both a photo and a PDF) kept hitting a 503
("high demand") — but plain-text `curl` calls to the same model succeeded every time,
confirmed via direct testing. This narrowed it to `gemini-3.5-flash`'s **image-processing
path specifically** being under real capacity pressure on Google's side, not a code bug, not
a transient blip retries alone would fix, and not specific to any one file (both a PDF and an
iPhone photo failed identically).

Built out `src/utils/gemini.ts` as a shared client for both features (previously
near-duplicated inline fetch code in each), with:
- **Image compression** before upload — downscales to max 1600px longest side, re-encodes as
  JPEG at 0.85 quality. PDFs pass through untouched (canvas can't rasterize them, and Gemini
  accepts `application/pdf` natively). Cuts payload size/cost with no OCR accuracy loss.
- **Retry logic** for transient 503/429 errors (2 retries per model, backing off).
- **`thinkingLevel: 'low'`** — discovered via testing that `gemini-3.5-flash` spends hidden
  "thinking" tokens on every call (115 thought tokens just to say "Hello" in one test) which
  cost money on top of visible input/output tokens. Can't be fully disabled on this model,
  but `'low'` avoids paying for reasoning depth a straightforward extraction task doesn't need.
- **Model fallback chain** — tries `gemini-3.5-flash` first, falls back to `gemini-2.5-flash`
  (presumably less demand-contended, being an older model) if the primary keeps 503ing,
  rather than failing the scan outright.

Both Invoice Scanner and Kitchen's recipe scan now use this shared client. Also swapped both
features' pantry-matching from a naive substring check to the shared `findBestIngredientMatch`
fuzzy matcher (same one Catalog/Pantry use) — handles this app's "Item - Descriptor" naming
(e.g. "Dill" now correctly matches Pantry's "Dill - Fresh"). Runs against the already-cached
`ingredients` query (5-minute staleTime shared query client) — **no new Firestore reads**,
which mattered given the whole session started because of a read-quota problem.

## Invoice Scanner bugs found via real-world testing (a real Booker invoice photo)
- **Qty/price doubling**: the prompt was ambiguous between an invoice's per-unit PRICE column
  and its extended VALUE column (PRICE × QTY). A line like "Philadelphia, QTY 2, PRICE 16.99,
  VALUE 33.98" was extracting `packCost: 33.98` instead of `16.99` — since Pantry pricing is
  always per-single-pack, this silently doubled (or worse) any multi-quantity invoice line.
  Fixed by making the prompt explicit about PRICE vs. VALUE with a worked example.
- **No way to fix an unmatched line**: added a "Link to Pantry Item" control on unmatched
  lines — a dropdown of all ingredients, picking one recomputes the current-price/diff fields
  identically to an automatic match, so it slots into Price Changes / No Change correctly.
  Expected to be needed often until the user finishes manually linking all ~3,000 supplier
  catalogue products to their canonical Pantry names (see `wip/catalogue-matcher` branch,
  parked from yesterday, built exactly for that job).
- **Fixed a related latent bug** while touching `handleUpdatePrice`: it previously blindly
  overwrote whichever supplier was `isPreferred` (or index 0) regardless of whether that
  supplier's name actually matched the invoice's supplier — for a freshly manually-linked
  ingredient with zero existing suppliers, this silently did nothing (mapping over an empty
  array). Now matches by supplier name first, falls back to preferred/index-0 for the common
  case, and appends a brand-new supplier entry if neither exists.
